- (decision) 파라다임: Pipes-and-Filters 오프라인 모델개발 파이프라인 + Layered 읽기전용 서빙(Frozen Artifact 소비). 밸류업 프로젝트와 동일 계열(서빙=Layered, 수집/처리=Pipes-filters) — 사용자에게 익숙한 패턴 재사용
- (decision) AD-1 Model Artifact Contract: 파이프라인과 API 사이의 유일한 통로는 버저닝된 아티팩트 번들(joblib 모델 + manifest.json: feature order, WOE bin edges, PDO/base score, model_version, trained_at). CAP-4,5,6,7,9,10,11,14,15 전부 이 번들을 쓰거나 읽는다
- (decision) AD-2 Train/Serve Parity: WOE 변환 로직은 scorecard/binning.py 단일 모듈에만 존재, 파이프라인 스크립트와 app/loader.py 둘 다 이 모듈을 import. API가 비닝 로직을 재구현하는 것을 금지 — 훈련/서빙 스큐 방지
- (decision) AD-3 Canonical Scored Validation Frame: 평가 단계(CAP-6,8)가 1회 생성하는 scored validation/OOT parquet(score,PD,grade,실제bad flag,int_rate,recoveries,vintage 포함)이 CAP-9,10,13,14,15의 유일한 데이터 소스. 각 캡이 독립적으로 예측을 재계산하지 않는다 — 대시보드·API 숫자 불일치 방지
- (decision) AD-4 Read-only Serving: app/(API 레이어)는 학습·재학습·아티팩트 변경을 절대 하지 않는다. 시작시 frozen 아티팩트+scored validation frame을 로드만 한다
- (decision) AD-5 [ADOPTED] Response Contract: API_SPEC.md(SPEC의 adopted companion)의 스키마가 구속력을 가진다. 필드 변경은 API_SPEC.md 선(先)수정 후 구현, /v1 내 추가만 허용, 파괴적 변경은 /v2
- (decision) AD-6 [ADOPTED] Reason Code Shape Parity: 챔피언·챌린저 reason_codes는 동일 rank/description 구조를 공유하고 값 필드만 다르다(points_lost vs shap_value) — 동일 pydantic 베이스 모델 상속으로 강제(SPEC CAP-11 계승)
- (decision) AD-7 [ADOPTED] Deterministic Rule-based Verdicts: rule_efficiency.py, profit.py의 판정/verdict 로직은 순수 규칙·통계 함수로만 구현, LLM이나 외부 API 호출 금지(SPEC 제약 계승)
- (decision) AD-8 Deployment Envelope: 로컬 단일 환경(dev only), uvicorn+streamlit 로컬 프로세스, DB 없이 parquet/파일 기반. 컨테이너화·클라우드 배포·CI는 이번 스코프 밖(P3 연동이 라이브로 필요해지는 시점에 재검토). 아티팩트는 models/artifacts/(gitignore) 로컬 디스크
- (decision) AD-9 Dependency Direction: app/(서빙)는 scorecard/(라이브러리)에 의존 가능, 역방향 금지. pipelines/는 scorecard/에 의존. dashboard/는 반드시 app/의 HTTP API를 통해서만 데이터를 가져오며 아티팩트 파일을 직접 읽지 않는다 — 대시보드와 API가 항상 동일한 숫자를 보여주도록 강제
- (question) 모형 재학습 주기·트리거는 이번 스코프에서 결정하지 않음(1회성 포트폴리오 산출물이므로) — Deferred
